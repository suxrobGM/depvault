# Error Codes

Every error response from the DepVault API follows a consistent JSON shape. This page documents all error codes, their meanings, and how to handle them.

## Error Response Format

All error responses share this structure:

```json
{
  "code": 400,
  "message": "Validation error",
  "details": "Optional additional context"
}
```

| Field | Type | Description |
|---|---|---|
| `code` | number | HTTP status code |
| `message` | string | Human-readable error description |
| `details` | unknown (optional) | Additional context, such as validation details. Only included in non-production environments for certain error types. |

The `code` field always matches the HTTP status code of the response.

## HTTP Status Codes

### 400 Bad Request

The request is malformed or fails validation.

**Common causes:**
- Missing required fields in the request body
- Invalid field format (e.g., malformed email address)
- Field value out of allowed range (e.g., password shorter than 8 characters)
- Malformed JSON in the request body

**Example response:**

```json
{
  "code": 400,
  "message": "Validation error",
  "details": "Expected string with at least 8 characters for 'password'"
}
```

**How to handle:** Check the `message` and `details` fields for specifics on what went wrong. Fix the request payload and retry.

### 401 Unauthorized

The request lacks valid authentication credentials.

**Common causes:**
- No access token cookie or Authorization header present
- Access token has expired
- Refresh token has been revoked or expired
- CI token is invalid or has been deleted

**Example response:**

```json
{
  "code": 401,
  "message": "Unauthorized"
}
```

**How to handle:** For browser clients, call `POST /auth/refresh` to get a new token pair. If refresh also fails, redirect to the login page. For CI tokens, verify the token is correct and has not been revoked.

### 403 Forbidden

The authenticated user does not have permission to perform this action.

**Common causes:**
- Attempting to delete a project you do not own
- Trying to manage members without the owner or admin role
- Accessing a project you are not a member of

**Example response:**

```json
{
  "code": 403,
  "message": "Forbidden"
}
```

**How to handle:** Verify the current user has the required role for the operation. Project owners can manage roles through the members API.

### 404 Not Found

The requested resource does not exist.

**Common causes:**
- Invalid resource ID (UUID)
- Resource has been deleted
- Accessing a route that does not exist

**Example response:**

```json
{
  "code": 404,
  "message": "Not found"
}
```

**How to handle:** Verify the resource ID is correct. If the resource was recently created, there may be a propagation delay. Check that the URL path is spelled correctly.

### 409 Conflict

The request conflicts with the current state of the server.

**Common causes:**
- Registering with an email that already has an account
- Creating a project with a name that already exists for the user
- Adding a member who is already part of the project

**Example response:**

```json
{
  "code": 409,
  "message": "A user with this email already exists"
}
```

**How to handle:** The `message` field describes the conflict. Adjust the request to avoid the duplicate. For registration conflicts, direct the user to the login flow instead.

### 410 Gone

The requested resource existed but has been permanently removed.

**Common causes:**
- Accessing a one-time shared secret link that has already been viewed
- Accessing a shared secret link that has expired

**Example response:**

```json
{
  "code": 410,
  "message": "Gone"
}
```

**How to handle:** The resource cannot be retrieved. For shared secrets, the sender must generate a new link.

### 429 Too Many Requests

The client has exceeded the rate limit for this endpoint.

**Common causes:**
- Too many login attempts in a short window
- Repeated registration attempts
- Automated scripts hitting rate-limited endpoints without backoff

**Example response:**

```json
{
  "code": 429,
  "message": "Too many requests, please try again later"
}
```

**Response headers:**

| Header | Value |
|---|---|
| `retry-after` | Number of seconds to wait before retrying |
| `x-ratelimit-limit` | Maximum requests allowed in the window |
| `x-ratelimit-remaining` | `0` |
| `x-ratelimit-reset` | Unix timestamp when the window resets |

**How to handle:** Wait for the number of seconds indicated by the `retry-after` header before retrying. See the retry strategy below.

### 500 Internal Server Error

An unexpected error occurred on the server.

**Example response:**

```json
{
  "code": 500,
  "message": "Internal server error"
}
```

**How to handle:** This indicates a server-side bug or infrastructure issue. The error message is intentionally generic to avoid leaking internal details. Retry the request after a short delay. If the error persists, contact support.

## Validation Errors

When the request body fails schema validation, the API returns a 400 response. In non-production environments, the `details` field includes specifics about which field failed and why:

```json
{
  "code": 400,
  "message": "Validation error",
  "details": "Expected property 'email' to be a valid email format"
}
```

In production, `details` is omitted to prevent information leakage. The `message` field still says `"Validation error"` so clients can distinguish validation failures from other 400 errors.

## Parse Errors

If the request body is not valid JSON, the API returns:

```json
{
  "code": 400,
  "message": "Malformed request body"
}
```

Verify that the `Content-Type` header is set to `application/json` and that the body is well-formed JSON.

## Retry Strategy

For transient errors (429, 500), use exponential backoff:

1. On a 429 response, read the `retry-after` header and wait that many seconds.
2. On a 500 response, wait 1 second, then retry.
3. Double the wait time on each subsequent failure (1s, 2s, 4s, 8s...).
4. Cap the maximum wait at 30 seconds.
5. Stop retrying after 3--5 attempts total.

```typescript
async function fetchWithRetry(url: string, options: RequestInit, maxRetries = 3) {
  let delay = 1000;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    const response = await fetch(url, options);

    if (response.ok) return response;

    if (response.status === 429) {
      const retryAfter = response.headers.get("retry-after");
      delay = retryAfter ? parseInt(retryAfter, 10) * 1000 : delay;
    }

    if (response.status === 429 || response.status >= 500) {
      if (attempt < maxRetries) {
        await new Promise((resolve) => setTimeout(resolve, delay));
        delay = Math.min(delay * 2, 30_000);
        continue;
      }
    }

    return response; // Non-retryable error, return as-is
  }
}
```

## Error Codes at a Glance

| Code | Name | Retryable | Description |
|---|---|---|---|
| 400 | Bad Request | No | Fix the request and resubmit |
| 401 | Unauthorized | Yes (refresh) | Refresh the token, then retry |
| 403 | Forbidden | No | User lacks required permissions |
| 404 | Not Found | No | Resource does not exist |
| 409 | Conflict | No | Duplicate resource conflict |
| 410 | Gone | No | Resource permanently removed |
| 429 | Too Many Requests | Yes (after delay) | Wait for `retry-after` seconds |
| 500 | Internal Server Error | Yes (with backoff) | Server-side issue, retry with backoff |

## Security Notes

Error messages are sanitized before being returned to clients:

- Sensitive values (passwords, tokens, encryption keys) are automatically redacted from error messages.
- Database-level errors are replaced with a generic "Internal server error" message.
- Stack traces are never included in production responses.

This prevents accidental information leakage through error responses.
